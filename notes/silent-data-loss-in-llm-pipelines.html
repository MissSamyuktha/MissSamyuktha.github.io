The silent data loss bug: 23 attributes where thousands should have been

A note from my experiments with LLM extraction pipelines. Names of internal tools omitted; the lesson is general.

The setup

I have been experimenting with a pipeline that turns unstructured company documents (filings, news, patents) into a knowledge graph. Two of its jobs matter for this story:

  1. Triple extraction: an LLM reads text chunks and emits relationship triples like (Company A) -[ACQUIRED]-> (Company B).
  2. Attribute extraction: a separate pass that captures financial and operational metrics (revenue, headcount, deliveries, capex) as attribute nodes.

Early versions of the triple extractor had a loose prompt. Among other sins, it emitted financial metrics as triples: (Tesla) -[reported]-> ($97.7B revenue). That is bad modeling. A revenue figure is not a relationship between two entities, and every downstream query has to special-case it. So when I did a quality-first rewrite of the extraction prompt, one of the explicit new rules was: do not extract financial metrics as triples. They belong to the attribute pipeline.

The rewrite worked. Triple quality jumped dramatically. I moved on.

The symptom

Weeks later, while checking the graph, I ran a count of attribute nodes for one company's full annual filing. The answer was 23.

A 10-K filing is a couple hundred pages dense with numbers. The attribute taxonomy had 20+ types defined. The expected volume was in the hundreds at minimum. 23 was not "the extractor is a bit conservative". 23 was "effectively nothing", quietly, with no errors logged anywhere. Every stage reported success.

The investigation

The attribute extractor's input was the thing I should have looked at first, but it took a while to get there, because the coupling was invisible in the architecture diagram. On paper, triple extraction and attribute extraction were independent stages.

In the code, they were not. The attribute extractor read the source_sentence field of previously extracted triples and looked for financial metrics in those sentences. That design had worked, but only by accident: the old, sloppy prompt emitted metric-shaped triples, so their source sentences were full of numbers, and the attribute extractor had plenty to chew on.

When the prompt rewrite correctly stopped emitting financial metrics as triples, there were no more numeric source sentences. The attribute extractor was still running, still succeeding, and receiving almost no input. It had been silently depending on a side effect of bad extraction. Fixing the bug upstream starved the pipeline downstream.

The fix

The fix was to make the architecture diagram true: decouple the two pipelines completely. The attribute extractor now runs directly on the raw prefiltered text chunks, with no dependency on triple extraction at all.

Since running an LLM over every chunk just to find numbers is wasteful, a cheap deterministic regex prefilter selects chunks that plausibly contain metrics before any LLM call: currency symbols, magnitude words like "billion" and "million", percentages, large numbers, and metric keywords (revenue, EBITDA, headcount, deliveries, and so on). Only those chunks go to the attribute extractor. 

_NUMERIC = re.compile(
    r'(\$|€|£|¥|CHF|USD|EUR|GBP|CNY)'
    r'|\b\d[\d,]*\.?\d*\s*(billion|million|thousand|bn|mn|m|b)\b'
    r'|\b\d+[\d,]*\s*%'
    r'|\b(revenue|EBITDA|margin|profit|capex|headcount|deliveries)\b',
    re.IGNORECASE,
)

numeric_chunks = [c for c in passed_chunks if _NUMERIC.search(c["text"])]
attr_results = extract_attributes(numeric_chunks)

Result on the same filing: 23 attributes became 758. A 33x improvement from one architectural change, with no new models and no prompt tuning.

What I took away from this

Pipelines coupled through data are still coupled. No import statement connected the two extractors. The dependency lived in the shape of the data one stage happened to produce. Those are the couplings that bite, because no diagram shows them.

Improving one component can starve another. The trigger for this bug was not a mistake. It was a correct fix. If two stages share data and one of them gets better at its job, the contract between them just changed, and nothing will tell you.

Silence is not success. Every stage returned green. The only signal was a number that was implausibly low, and I only saw it because I happened to count. Since then I treat expected-volume checks ("a 200-page filing should yield hundreds of attributes, alert if under 100") as part of the pipeline, not as ad-hoc debugging.

Cheap deterministic filters earn their keep. A regex is not glamorous next to an LLM, but it cut the candidate set drastically before any tokens were spent, and it is fully predictable. The best LLM pipelines I have built so far are mostly deterministic plumbing with the LLM doing only the part nothing else can do.
